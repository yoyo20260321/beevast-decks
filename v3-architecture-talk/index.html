// part 01 · why

V2 撑不住了 —— V3 要解的是协作和产品化

先讲为什么要做 V3。V2 跑了一段时间，不是性能不够，是协作和产品化撑不住。

左边五条是 V2 的核心痛点。最扎眼的两条：memory 各自一份——bwg 在 worktree A 教过 agent 的事，到 worktree B 就忘了；skill / subagent 硬编码——加一个 skill 要改代码 commit push 上线。

右边是 V3 的对应解。workdir 共享 / worktree 独立这条是核心——后面有整整一页讲。tarball + 一键 install 是产品化的入口，客户机一行命令装好。

所以 V3 解的不是"跑得更快"，是"3 个开发者 + N 个客户 + 多产品"这件事跑得起来。下面进总览图。

这是我们五层架构的总览。从下往上看更顺：

这是过去两周拓扑的重大变化，也是要重点讲的一页。原来一个 master-brain bot 一把抓——既接 IM、又跑 agent、license 也自己读自己。现在拆成 4 个产品。

im-gateway：以前 IM 接入耦合在 master-brain 里，现在抽出独立 product。一个 IM bot 一个 im-gateway 进程，挂在企业 gatekeeper 上往后转发。新加渠道不动 master-brain。

gatekeeper：每个企业部署一套（on-prem），是「用户↔可见 agent」「license 履约」的真相源，自带控制台。RFC 0028。

master-brain / analyst / ops rpc：一进程一 agent，各自启 profile，各自挂 RPC 端口。im-gateway 经 gatekeeper 路由到对应 agent 的 RPC。

beekeeper：中心化 license CA。客户机不下发私钥，只下发签名后的 license 文本——这是 RFC 0029 的信任锚。

关键: "产品" 这个词承载的是边界。打包、部署、升级、命名空间各自独立。某一个挂了不连累全局。

// part 02b.5 · 数据流转大图

从用户消息到 agent 回复 —— 启动期 3 跳 · 运行期 8 跳

同一张图看完: 启动期 license/注册 (虚线 A/B/C) + 运行期消息处理 (实线 ①-⑧)。

这一页把从用户消息到 agent 回复的完整链路一次性讲清楚。重点在于看清启动期和运行期各自做了什么。

启动期有 3 跳，图上虚线 A/B/C：

A — 每个 agent profile-start 起来后会调 gatekeeper /authz/register 注册自己（带公钥 + rpcUrl），首次默认 pending 等 admin 批准；已批准则 active；之后周期心跳保活。这就是 RFC 0027 §审批制。

B — im-gateway 启动后调 gatekeeper /authz/agents 拿到所有在线 agent 的目录（含 rpcUrl 映射），后续每来一条消息都会再校一遍。

C — gatekeeper 启动会调 beekeeper /v1/activate 用激活码换 license + credential；之后周期 /v1/refresh 保持 license 新鲜，/v1/report 上报用量心跳。

运行期是 8 跳，图上实线 ①-⑧：

① 用户从任一渠道发消息 → ② im-gateway 调 gatekeeper resolve 拿 principalId + 可见 agent 集 + license 状态 → ③ 回来。

④ im-gateway 按 per-chat 选定的 agent（或 /use 切换的）转发到对应 RPC /brain/prompt。

⑤ RPC 鉴权闸再次调 gatekeeper resolve 二次校验 + 同步 fire-and-forget 调 /authz/usage 埋点。

⑥ 进 BeevastRuntime → claude ACP child 处理 → ⑦ 把 RuntimeEvent 通过 SSE 流式回给 im-gateway → ⑧ im-gateway 用对应渠道的 Renderer 渲染（飞书 patch 卡片 / web SSE / 微信 buffer 整段）发回原渠道。

关键观察: 用户感知不到中间这 5 层，整个链路对用户透明；但任何一层异常都能定位（gatekeeper 不通 = 拒鉴权 / beekeeper 不通 = 离线宽限 / agent RPC 不通 = "助手不在线"）。

// part 02c · RFC 0027 + 0028 拓扑修订

鉴权闸下沉到 agent RPC —— 唯一入口

从"每个渠道各自调闸"改成"agent RPC 自带闸"。新渠道接入零改造、漏不掉。

🔴 老路径 · 边缘鉴权

飞书 bot → 自己调 gatekeeper

钉钉 bot → 自己调 gatekeeper

web / 桌面 → 自己调 gatekeeper

问题：新渠道接入必须记得加这一步；漏一个就漏鉴权、漏统计、漏埋点。

安全口子：桌面端若直连 agent RPC 之前不经鉴权（仅 loopback 拦着）。

🟢 新路径 · 统一闸

所有渠道 → im-gateway / web 前端

→ 转发到 agent RPC /brain/prompt

→ RPC 处理前调 gatekeeper resolve
　 + fire-and-forget 埋点 /authz/usage

优势：新渠道接入零改造；任何客户端走 RPC 都被鉴权 + 统计。

边缘瘦身：web/dispatcher 的 resolve 退化为列出可见 agent给下拉框，不再做强制。

唯一入口 = 唯一防护点。认人 + 鉴权 + 埋点统一在 agent RPC 闸一处实现，新渠道接入不会漏。POST /authz/usage 落 gatekeeper usage_events，控制台「数据」tab 聚合显示活跃自然人/会话/调用。

RFC 0027 + 0028 联动的核心修订: 鉴权执行点从边缘下沉到 agent RPC /brain/prompt。

这是 bwg 2026-05-26 复盘后定的方向，不是 RFC 原计划。原计划在每个渠道（web 前端 / IM dispatcher）各自调 gatekeeper 判权限。改完后整条链路只剩一个执行点。

RPC 收到 prompt 前先调 gatekeeper resolve(platform, channelUserId)，校验「本 agent 是否对该用户可见 + license 是否有效」，命中后回填解析后的自然人身份注入 runtime（给 agent 记喜好），不通过 403 直接拒。

顺手解决一个安全漏洞：桌面端 ws-server 之前直连 agent RPC 不经 gatekeeper，仅靠 loopback 拦。现在统一过闸。

边缘的 resolve 退化为纯展示用途：列出用户可见的 agent 填下拉框，不再强制。

// part 02d · RFC 0029 信任模型

beekeeper —— License 中心 + 防伪造信任锚

on-prem 客户有 root → 可以改配置指向伪造 gatekeeper。唯一伪造不了的 = beekeeper Ed25519 签名。

📜 gatekeeper · 运营 / 软

持 credential 连 beekeeper

激活 / 刷新 / 缓存 / 离线宽限

控制台展示 + 软限 IAM 操作

N 个 agent 不该各自连 beekeeper, 集中在这

客户能控制 → 不当裁决者

⚖️ agent RPC 闸 · 裁决 / 硬

不直连 beekeeper

内嵌公钥自验签 license 原文

硬执行 agents_enabled / 到期(+宽限) / 吊销暂停

信 gatekeeper, 但要求"真 gk"证明

信任锚 = beekeeper 私钥的 Ed25519 签名

三道防护: ① agent 内嵌公钥自验签 license — 客户伪造 license 直接挡 ② gatekeeper 出示 beekeeper 签发的身份证书证明"真 gk" — 防 url 替换 ③ 异常事件上报中心 + 黑名单拉黑伪造源 — 防长期渗透。

RFC 0029 解决一个尖锐问题: 客户机有 root，把 agent 的 [gatekeeper] url 指向自己伪造的 gatekeeper 怎么办?

伪造 gatekeeper 可以一律返回「license 有效 + 所有 agent 可见」，绕过席位 / agents 白名单 / 到期 / 吊销这些商业边界。这是 on-prem 部署的天生威胁。

解法 = 分工 + 三道防护：

左边 gatekeeper（软）：留运营功能——连中心、缓存、离线宽限、给企业管理员看 dashboard。但不让它当裁决者。它的结论 agent 不全信。

右边 agent（硬）：内嵌 beekeeper 公钥，自己验签 license 原文，硬执行 agents_enabled / 到期 / 吊销。这是裁决权所在。

三道防护逐层挡：① agent 验签挡伪造 license ② gatekeeper 出示 beekeeper 签发的身份证书挡 url 替换 ③ 异常上报 + 中心黑名单挡长期渗透。

核心理念: 客户伪造不了 Ed25519 签名——所有硬执行都锚在签名上，不锚在某个可被客户控制的服务的结论上。

// part 03 · innovation #1

Agent workdir ≠ Code worktree

两个目录、两条生命周期、绝不合并。

🟢 Agent workdir（共享）

路径：~/.beevast/products/master-brain/envs/<env>/workdir/

承载：AGENTS.md / .agents / .claude / memory（全是 symlink）

隔离：env 内所有 chat 共享

作用：claude ACP child 的 spawn cwd · system prompt · memory 单源

🟢 Code worktree（隔离）

路径：~/git/beevast-trees/<bot>-<hash8>/

承载：完整 monorepo + 独立分支

隔离：每个 chat（routeKey）各一份

作用：真改代码 / git commit / 多任务并行不撞

双隔离 + 双共享：workdir 共享 → memory/skill 跨 chat 一致；worktree 隔离 → 多任务 git 不污染。envelope.code_workdir 告诉 agent 代码在哪、ACP child cwd 始终是固定 agent workdir。

这是 V3 最容易踩坑的地方，我自己上周就刚踩过——这页是核心创新一。

左边 Agent workdir：路径在 envs/dev/workdir/，里面全是软链接——AGENTS.md / .agents / .claude / memory，统统指向 shared 仓。这个目录是所有 chat 共享的。你在 chat X 让 agent 写下经验，chat Y 立刻读得到，跨 worktree 一致。

右边 Code worktree：路径在 ~/git/beevast-trees/feishu-dev-bwg-ec71c6cb/ 这种带 chat_id 哈希的目录，每个 chat 一份。你在 chat X 改 feat-A、在 chat Y 改 fix-B，互不污染。

下面那个底框是容易讲错的——很多人会把两个目录搞混。一句话：agent 跑在哪是 workdir、改的代码在哪是 worktree。两个永不重合（prod 例外，prod 没有 per-session worktree）。

// part 03 · innovation #2

可配置的 Agent 编排 —— agent.toml

agent 能力（skill / subagent / rules / memory / permissions）全配置驱动，不进代码。

# ~/.beevast/shared/agents/master-brain/agent.toml [agent] name = "master-brain" agents_md = "./AGENTS.md" [mount.skills] sources = [ "shared://skills/ops/master-brain-deploy", "shared://skills/ops/snapshot", "shared://skills/deliverable/html-ppt", # ← 昨天刚加 "plugin://superpowers-dev/brainstorming", ] [mount.subagents] extends = [ "shared://subagents/_packs/essential" ] [mount.memory] source = "memory://agents/master-brain" auto_pull = true auto_push = true [permissions] template = "admin-full"

核心创新二——可配置编排。这是 master-brain agent 的 spec 文件，三十几行的 TOML 就把 agent 的能力定死了。

看 [mount.skills] 这一段——agent 能用哪些 skill 全列在这里。scheme 三种：shared:// 是 beevast-shared 仓里的、plugin:// 是 Claude Code 插件市场的、project:// 是当前 monorepo 的。我昨天加 html-ppt skill，只动了这一行——不改代码、不重启 bot、跑一次 project-agent 命令立刻生效。

下面 [mount.subagents] 用 extends 引一个 starter pack，加几个特定的。[mount.memory] 配独立仓 + auto-pull/push。[permissions] 走模板。所有动作都是声明式的，不写 if/else。

V2 时这套全埋在代码里，加一个 skill 要 commit + PR + 重启。V3 改 toml + 跑一行命令。

// part 03 · innovation #3

Shared 资源单源 —— beevast-shared 仓

agent 定义 / skill / subagent / rules / 模板 —— 全部抽到独立仓库。

  • 一处更新，全 worktree 自动同步（软链而非拷贝）
  • 跟代码解耦（不需要等代码 PR 合）
  • 跨产品复用（master-brain / weimao 共享同一套 ops skill）
  • 多人并行无冲突（worktree 各跑各的代码，shared 资源同步）
# 投影机制（agent-loader） $ ls -la envs/dev/workdir/.claude/skills/ master-brain-deploy → ~/.beevast/shared/skills/ops/... snapshot → ~/.beevast/shared/skills/ops/... html-ppt → ~/.beevast/shared/skills/deliverable/... brainstorming → ~/.claude/plugins/.../brainstorming # 改 shared skill 后 $ pnpm tsx products/master-brain/src/bin/project-agent.ts \ --workdir ~/.beevast/products/master-brain/envs/dev/workdir \ --agent master-brain [project-agent] ✓ projection complete

核心创新三——shared 资源抽成独立仓。这件事的杀手锏在并行开发的 worktree 也能保持一致。

左边四条收益。最有体感的是跨产品复用——master-brain 用的 ops skill，weimao 也能用，不需要 fork 一份。所有产品都从 ~/.beevast/shared/ 投影。

右边是真实命令。.claude/skills/ 目录里每个 skill 都是软链，不是拷贝。所以你改 shared 仓里的 skill 文件，不需要重启任何 bot，下一次 agent 读 SKILL.md 时就拿到新版。

下面那个 project-agent 命令是幂等的——可以反复跑，跑完所有 worktree 都对齐到 shared 仓最新状态。我昨天加 html-ppt skill 就是跑了这一行命令。

🟢 协作场景

多位开发者在不同 worktree 跟 agent 对话 → 所有 memory 落在同一个仓。bwg 教过的事，Tim / lll / ydl 的 agent 自动读得到。

🔵 演进方向

短期：单 git 仓 + auto pull/push。中期：加向量检索（语义召回 + 减少 prompt token）。长期：独立知识库服务 + 团队订阅。

核心创新四——Memory 独立仓。这条乍看跟 shared 仓很像，但语义完全不同。

Shared 仓是团队预定义资产——skill / rules 这些事先准备好的东西。Memory 仓是agent 运行时沉淀——每次跟我对话学到的偏好、踩过的坑、项目状态。每个 agent 都有 auto-pull / auto-push，session 启动拉一次、关键时间点 push 一次。

下面左边讲协作价值——这是当前最大的红利。bwg 教 agent 的事（比如飞书消息格式紧凑），到 Tim 那边的 agent 也能读到，团队共享认知。

右边讲未来——现在是 git 仓 + auto-sync，下一步加向量检索，agent 不用每次塞 100KB memory，按相关度召回最近的 5 条就行。长期可能拆成独立服务，多团队订阅。

核心创新五——产品化分层 + 客户打包。

左边讲分层。最底下 packages/ 是公共下沉的 npm 包，acp-runtime / gateway / renderer 这些跨产品共用。services/ 是共享 service。tools/ 是 CLI（dispatcher / packer）。products/ 在最上层——master-brain / weimao / website 之间平行不互依赖，但都可以依赖下游 packages。这条规则保证产品边界清晰，不会出现"主脑里 import 了官网代码"这种乱搞。

右边讲打包流程。pack-v3 把 dist + launch profiles 打 tarball。客户机上 beevast install 拉 tarball 解压。然后 bots.ts add 写凭证（昨天刚把 RFC 0015 文档统一了），最后 beevast start 跑起来。四条命令客户机就跑起来了，不需要客户懂 monorepo / pnpm / workspace 这些。

进入第二大块——五层逐层细节。先讲 L3 和 L2，因为它们是入口。

上面那张流程图：IM 平台原始 frame 进 L3 → 归一化成 InboundEnvelope（自带 UUIDv7 traceId）→ L2 按 routeKey 把它路到对应会话。

左边 L3 Gateway。两个 adapter：WsAdapter 给 ws transport 用，ChannelBridgeAdapter 是个"包装层"——它把 V2 已经写好的 feishu / wecom / dingtalk 代码包起来不重写，这条原则救了我们大量工时，feishu 那 1200 多行卡片渲染逻辑没动一个字。

右边 L2 Session Conductor。routeKey 是会话标识符，默认 platform:chat_id，可以在 profile 里改模板。最值得提的是L1Error 归类——L1 抛出的所有错误都被归到三类：用户能看到的、可以重试的、致命的，L4 据此决定要不要发给用户。

// part 04 · layer detail · L1

L1 · Agent Runner (核心)

1 bot daemon (Node process) └─ ConversationManager // LRU map: routeKey → entry, max 1000, 30min idle evict └─ ConversationEntry (per routeKey, lazy on first inbound) └─ 1 AcpAgent └─ spawn 1 个 claude-agent-acp 子进程 └─ 内含 1 个 ACP session └─ cwd = envs/<env>/workdir/ // 固定 agent workdir!
  • LRU + idle evict：保活高频会话、清掉超时的、磁盘 + token 不爆
  • per-routeKey 隔离：每个 chat 独占一个 claude 子进程 + ACP session
  • resume 逻辑：acp-sessions.json 持久化 routeKey → sessionId，重启续上
  • token / cost 事件：每个 turn 结束发 token.usage + cost.accrued

L1 是整个 V3 最关键的一层，今天讲细一点。

看上面这个嵌套结构。一个 bot daemon 进程内含一个 ConversationManager——它是个 LRU map，key 是 routeKey，最多 1000 个 entry，30 分钟没活动的会被 evict。每个 entry 里有一个 AcpAgent，AcpAgent 又 spawn一个独立的 claude-agent-acp 子进程。所以同一个 bot 下不同 chat 是真物理隔离——不同进程、不同 session、不同 cwd。

注意最底下那行——cwd = envs/<env>/workdir/，固定 agent workdir。这就是回到 #4 那页讲的——不管哪个 chat，ACP child 的 cwd 都是同一个共享 workdir，memory 才能跨 chat 一致。

四个 bullet：LRU + idle 控资源、隔离防串、resume 撑重启、cost 事件给可观测。这一层做对了，后面 7 个开发者并行跑都不会乱。

L0 Loader · L4 Renderer · Launch Profile

L0 · Agent Loader

agent.toml → 投影到 workdir（symlink）

workdir-lock 用 lockSuffix=profile.meta.name → 4 个 channel profile并发共享同一 workdir

校验 + dry-run + force overwrite

L4 · Renderer

RuntimeEventEnvelope 流（text_chunk / tool_call / turn_end）

channel-aware：feishu 走 ProcessCardRenderer（streaming PATCH 卡片）、ws 走真 streaming、wechat 走 buffer

role-aware：admin 看完整 trace、普通用户看精简

Launch Profile

profile TOML extends 链 + [env.<label>] overlay

凭证不进 profile，从 bots.json 反查注入（RFC 0015）

limits / permissions / routing 全配置驱动

剩下三块一起讲，因为它们都是支撑性的横切。

左边 L0 Agent Loader——把 agent.toml 投影成 workdir 软链。最关键的设计是 workdir-lock 加了 lockSuffix，比如 lockSuffix 用 profile 名（feishu-dev-bwg）—— 4 个 channel profile 同时跑同一个 master-brain workdir 不会撞锁，多 IM 并发是这里解的。

中间 L4 Renderer——把 L1 流出来的事件按平台特性渲染。feishu 走流式 patch 卡片那条路径，ws 走真 SSE，wechat 没法流式只能 buffer 整段发。这层把渠道差异封掉，L1 不用管。

右边 Launch Profile——一个 bot 的所有配置在这里。注意凭证从不进 profile，profile 只写 appId 这种公开 ID，secret 在加载阶段从 bots.json 反查注入，这就是昨天我们刚统一文档的 RFC 0015。

第一，双隔离双共享——workdir 共享让 memory 跨 chat 一致，worktree 隔离让代码不互撞。第二，配置驱动——agent 能力在 toml 里，不再写代码 commit。第三，shared 单源——一处更新全 worktree 自动同步。第四，memory 独立仓——团队认知集中沉淀，向知识库演进。第五，产品化 + 一键装——客户四条命令跑起来。
